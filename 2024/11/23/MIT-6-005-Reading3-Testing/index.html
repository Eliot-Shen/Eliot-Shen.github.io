MIT 6.005 Reading3 Testing

  • understand the value of testing, and know the process of test-first programming;
  • be able to design a test suite for a method by partitioning its input and output space and choosing good test cases;
  • be able to judge a test suite by measuring its code coverage; and
  • understand and know when to use blackbox vs. whitebox testing, unit tests vs. integration tests, and automated regression testing.

Validation

测试你的程序正确性—即为：验证Validation，发现程序中潜在的bug并增强其可信度。

验证有三种方式：

  1. 形式化证明。用正式的证明方式来验证程序的正确性。手工证明很难，而自动化验证工具领域近些年来很活跃。对于小型程序可以用手工方式验证，比如调度器，字节解释器，文件系统等。

  2. Code review. 让别人来审阅你的代码，你的同伴，学长，或更有经验的人。

  3. Testing. 通过设计输入输出对，老老实实对你的程序做测试。

验证难以企及完美。可以用bug/千行代码的指标来估计，以下是几个例子：

  • 1 - 10 defects/kloc: 典型的工业界软件
  • 0.1 - 1 defects/kloc: Java通用库
  • 0.01 - 0.1 defects/kloc: NASA

Why Software Testing is Hard

受限于精度范围，比如int，float范围，没法在数域上做到穷尽。

随意选取的测试案例并不能找到什么bug，也不严谨。

统计学测试并不适用于软件领域。软件的缺陷在时空上并不是连续的，而是离散的，经常在边界点崩溃。

现在，作为一个测试者，你需要转变心态，承认自己程序存在潜在错误，并积极去找出它！

Test-first Programming 测试优先编程

Test early and often.

Don’t leave testing until the end, when you have a big pile of unvalidated code.

尽早开始测试，不要留到最后，面对一堆长长的代码，将大大增加测试的难度。

The development of a single function proceeds in this order:

  1. Write a specification for the function.
  2. Write tests that exercise the specification.
  3. Write the actual code. Once your code passes the tests you wrote, you’re done.

specification 描述函数的输入和输出，给出参数的类型及其约束(eg:非负性)，返回值。

Choosing Test Cases by Partitioning

将函数的输入空间划分为几个互不相交的子区域(函数的一组输入)，子区域一起覆盖了函数的整个输入空间，从子区域上各随机取点作为测试案例代表该区域，加起来，就变成了测试集。
这种方法既节省了测试资源，又能涵盖函数的所有情况，节约高效。

Example: BigInteger.multiply()

我们可以按a,b的正负性加特殊值(eg: 1，0，-1)进行分区。
我们还需思考案例本身的性质，比如这是一个大整数乘法，我们需要测试当ab非常大的情况下，算法是否还能正常工作，所以还需按大小分区。

最终划分得到的区域如下：

  • 0
  • 1
  • -1
  • small positive integer
  • small negative integer
  • huge positive integer
  • huge negative integer

覆盖了a和b组成的所有输入空间，接下来从每个区域选取一个点作为代表点.
(a,b) = (-3, 25) to cover (small negative, small positive)

(a,b) = (0, 30) to cover (0, small positive)

(a,b) = (2^100, 1) to cover (large positive, 1)

etc.

第二个例子：max()

max : int × int → int

该函数有三种关系：a < b，a = b，a > b

所以我们的测试集应为：

(a, b) = (1, 2) to cover a < b

(a, b) = (9, 9) to cover a = b

(a, b) = (-5, -6) to cover a > b

Include Boundaries in the Partition

bug经常发生在边界条件上发生。

例子：

  • 0是正负数的边界
  • int和double的最小和最大边界值
  • 空的字符串，数组，列表等等集合类型
  • 集合类型的第一个和最后一个元素

为什么bug经常发生在边界条件上发生？

  1. 因为程序员经常犯“偏移一位”的错误。
    例如将<=写成<，将计数变量counter初始化为0而不是1。
  2. 这些边界可能需要作为代码中的特殊情况来处理。
  3. 边界上导致数值上的不连续性，比如int变量过大，从正数溢出至负数。

所以我们要记得将边界包括在测试分区内，从边界选取值作为测试输入。

eg: max : int × int → int

可以分区为：

  • relationship between a and b

a < b

a = b

a > b

  • value of a

a = 0

a < 0

a > 0

a = minimum integer

a = maximum integer

  • value of b

b = 0

b < 0

b > 0

b = minimum integer

b = maximum integer

接下来我们从这些分区中选择测试用例:

(1, 2) covers a < b, a > 0, b > 0

(-1, -3) covers a > b, a < 0, b < 0

(0, 0) covers a = b, a = 0, b = 0

(Integer.MIN_VALUE, Integer.MAX_VALUE) covers a < b, a = minint, b = maxint

(Integer.MAX_VALUE, Integer.MIN_VALUE) covers a > b, a = maxint, b = minint

Two Extremes for Covering the Partition

分区有两个极限情况:

  1. 完全笛卡尔乘积

每一种代表测试元素的组合，可以覆盖整个测试空间。

  1. 每一个分区被至少覆盖一次

实际测试中常常在两者中取折中。

Blackbox and Whitebox Testing

回忆”specification”是函数的行为说明:参数类型，返回值，参数限制和彼此间关系等等

黑盒测试指只根据specification来挑选测试用例，而不关心函数的具体执行。

白盒测试指根据函数体的具体执行来挑选测试案例，eg:函数体使用了不同的算法来完成同一件事，那么你就需要选取适用于测试不同算法的不同测试案例。如果函数缓存记录了之前的输入，那么你也许就需要重复测试案例来测试功能。

但请注意:你的测试用例不需要尝试规格说明中没有明确要求的实现行为。例如，如果规格说明中说“如果输入没有格式化，那么将抛出异常”，那么你不应该特地的检查程序是否抛出NullPointerExpection异常，因为当前的代码实现决定了程序有可能抛出这个异常。

练习:

对于start参数，请选取一个测试分区:

A. start = 0, start = 5, start = 100

B. start < 0, start = 0, start > 0

C. start = 0, 0 < start < text.length(), start = text.length()

D. start < text.length(), start = text.length(), start > text.length()

答案为C

A没有测试边界值

B没有测试边界值且测试了非法值

C覆盖了start的合法范围，测试充分

D只测试了边界值

Documenting Your Testing Strategy

我们需要记录我们的测试策略，以便日后审查。

测试策略包括:我们是如何分区的，有哪些特殊值和边界值要考虑。

Coverage

评估一个测试集的好坏还有一个指标就是:覆盖率。

常见的三种覆盖:

  • 声明覆盖率: 每一个声明都被测试到了吗？
  • 分支覆盖率:对于每一个if 或 while 等等控制操作，它们的分支都被测试过吗？
  • 路径覆盖率: 每一种分支的组合路径都被测试过吗？

其中，分支覆盖率要比声明覆盖率严格（需要更多的测试），路径覆盖率要比分支覆盖率严格。在工业界，100%的声明覆盖率一个普遍的要求，但是这有时也是不可能实现的，因为会存在一些“不可能到达的代码”（例如有一些断言）。100%的分支覆盖率是一种很高的要求，对于军工/安全关键的软件可能会有此要求 (e.g., MC/DC, modified condition/decision coverage)。不幸的是，100%的路径覆盖率是不可能的，因为这会让测试用例空间以指数速度增长。

一个标准的方法就是不断地增加测试用例直到覆盖率达到了预定的要求。在实践中，声明覆盖通常用覆盖率工具进行计数。利用这样的工具，白盒测试会变得很容易，你只需要不断地调整覆盖的地方，直到所有重要的声明都被覆盖到。

在Eclipse中有一个好用的代码覆盖率工具 EclEmma 。如上图所示，EclEmma会将被执行过的代码用绿色标出，没有被执行的代码用红色标出。对于一个分支语句，如果它的一个分支一直没有被执行，那么这个分支判断语句会被标为黄色。例如上图中，我们发现for循环中的if语句每一次都是假的，我们下一步要做的就是调整测试用例使得这个判断可以为真。

Unit Testing and Stubs

一个好的测试程序应该对每个独立的模块有测试，这叫做单元测试。

其对立面就是集成测试，测试一组集成的模块，甚至是整个程序。但你不能只有集成测试，否则出错时就需要去寻找bug，它可能在程序的任何地方。集成测试仍然很重要，因为模块之间的交互很有可能出错，比如交互时的数据类型不匹配。

Example：
下面假设你在设计一个搜索引擎。其中有两个模块 getWebPage(), extractWords() ，其中 getWebPage()负责下载网页，extractWords() 负责将页面内容拆成一个个词汇。

而这两个模块又是被另一个模块 makeIndex()作为网络爬虫的一部分使用的。

我们的测试可以分为：

  • 对getWebPage()进行单元测试，输入不同的 URLs
  • 对extractWords()进行单元测试，输入不同的字符串
  • 对makeIndex() 进行单元测试，输入不同的 URLs

测试员有时会犯这样一个错误：extractWords() 的测试用例依赖于getWebPage() 的正确性。正如前面所提到的，单元测试应该尽可能将模块孤立起来。如果我们在对extractWords() 的测试输入进行分区后，其值中包含有 getWebPage() 的输出，那么如果getWebPage() 本身就有bug，程序的测试将变得不可信！正确的做法应该是先组建好独立的测试用例，例如一些下载好的网页，将其作为测试用例进行测试。

注意到 makeIndex() 的单元测试并不能完全孤立，因为我们在测试它的时候实际上也测试了它调用的模块。如果测试失败，这些bug也可能来自于它调用过的模块之中——这也是为什么我们要先单元测试 getWebPage() 和 extractWords() ，这样一来我们就能确定bug出现在链接这些模块的代码之中。

综上所述，测试时需要考虑测试模块之间的依赖关系，并以此来决定测试顺序。

如果我们要做更高于 makeIndex()这一level的测试，我们将它调用的模块写成桩 。例如，一个 getWebPage() 的桩不会真正去访问网页，而是返回一个预先设置好的网页（mock web page），不管参数URL是什么。一个类的桩通常被称为“模拟对象”（ mock object）。在构建大型系统的时候桩是一种重要的手段，但是在本门课程中我们不会使用它。[这里的Stubs桩，个人理解为模块的占位符]

Automated Testing and Regression Testing

自动化测试，即自动运行测试并给出测试结果，无需与程序员交互，无需程序员在过程中给出输入。自动化测试驱动会自主用固定的测试用例测试模块，并检查结果。

但即使自动化测试工具比如JUnit很方便，我们也应该自己想出好的测试用例，自动生成测试用例仍是一个前沿领域。

“””
要特别注意的是，当你修改你的代码后，别忘了重新运行之前的自动化测试。软件工程师经常遭遇修改大型/复杂程序所带来的痛苦，不论是修改一个bug、增加一个新的功能、优化一段代码的性能，都有可能带来新的问题。无论什么时候，自动化测试都能保证软件最重要的底线——行为和结果是正确的，即使只是一小段测试。我们称修改代码带来新的bug的现象为“回归”，而在修改后重新运行所有的测试称为“回归测试”。

一个好的测试应该是能发现bug的，你应该不断的充实你的测试用例。所以无论什么时候修改了一个bug，记得将导致bug的输入添加到你的测试用例里，并在以后的回归测试中去使用它——毕竟这个bug已经出现了，说明它可能是一个很容易犯的错误。

这些思想也是“测试优先debugging”的核心，当bug出现时，立刻将触发bug的输入存放到测试用例中，当你修复bug后，再次运行这些 （译者注：注意不仅是触发bug的输入）测试，如果它们都通过的话，你的debug也就完成了。

在实践中，自动化测试和回归测试通常结合起来使用。因为回归测试只有自动化才可行（不然大量的测试没法实现）。反过来，如果你已经构建了自动化测试，你通常也会用它来防止回归的发生。所以自动化回归测试(automated regression testing)是软件工程里的一个“最佳实践”（best-practice）。
